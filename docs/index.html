Overview

Pi is a terminal-based AI coding agent by Mario Zechner. Out of the box it provides tool use, conversation memory, a TUI, and an extension API.

agent is a set of 28 extensions, 6 agent definitions, 11 themes, and configuration files that transform Pi into a multi-agent orchestration platform with:

  • 6 operational modes — NORMAL, PLAN, SPEC, PIPELINE, TEAM, CHAIN
  • Multi-agent orchestration — dispatch teams, run sequential chains, or execute parallel pipelines
  • Security hardened — pre-tool-hook guard blocks destructive commands, detects prompt injection, prevents data exfiltration
  • Browser-based viewers — plan approval, completion reports with rollback, spec review with inline comments
  • 30+ model providers — Anthropic, OpenAI, Google, Mercury, Synthetic, OpenRouter
  • Task discipline — the agent must define tasks before tools unlock; three-state lifecycle with Commander sync

Everything is configuration — no forks, no patches. Just extensions, agent definitions, and YAML.

Quickstart

Prerequisites

  • Bun (v1.0+)
  • Pi Coding Agent installed globally

Install

# Clone the repo
git clone https://github.com/ruizrica/pi-dev.git
cd pi-dev

# Install dependencies
npm install

# Run Pi with all extensions
cd agent && pi

Pi reads agent/settings.json on startup, which loads all 28 extensions automatically.

First Steps

  1. Type a task — Pi operates in plan-first mode. It asks you to define tasks before using tools.
  2. Shift+Tab — Cycle operational modes
  3. Ctrl+T — Cycle themes
  4. /agents-team — Switch agent teams
  5. /chain — Switch chain workflows
  6. /security status — Check security posture

Extension Loading

Extensions are listed in the packages array in settings.json. Each is a relative path to a TypeScript file that exports a default function. Pi loads them in order at startup.

How Extensions Work

Each extension receives the Pi ExtensionAPI and can register:

  • Tools — LLM-callable functions with typed parameters and execute handlers
  • Commands — Slash commands (/foo) with autocomplete and handler functions
  • Shortcuts — Keyboard bindings (ctrl+k, shift+tab)
  • Widgets — TUI components rendered above or below the editor
  • Hooks — Lifecycle callbacks (tool_call, context, before_agent_start, etc.)

Multi-Agent Orchestration

agent provides three complementary orchestration patterns. Each uses a different extension and configuration format.

Teams

Teams are defined in agent/.pi/agents/teams.yaml. Each team is a named list of agent roles. In TEAM mode, the primary agent has no codebase tools — it can only delegate work to team members via the dispatch_agent tool.

# teams.yaml
plan-build:
  - planner
  - builder
  - reviewer

quality:
  - reviewer
  - tester
  - red-team

Agent definitions live in agent/.pi/agents/*.md with YAML frontmatter specifying name, description, and allowed tools. Each dispatched agent gets its own Pi session for persistent memory.

Chains

Chains are sequential pipelines defined in agent/.pi/agents/agent-chain.yaml. Each chain is a sequence of steps where each step specifies an agent and a prompt template. Output from step N becomes $INPUT for step N+1. $ORIGINAL always refers to the user's original prompt.

# agent-chain.yaml
plan-build-review:
  description: "Plan, implement, and review"
  steps:
    - agent: planner
      prompt: "Plan the implementation for: $INPUT"
    - agent: builder
      prompt: "Implement the following plan:\n\n$INPUT"
    - agent: reviewer
      prompt: "Review this implementation:\n\n$INPUT"

Security System

agent's security system operates at three layers, implemented by the security-guard extension.

Layer 1: Tool Call Gate

The tool_call hook fires before every tool execution. It blocks:

  • Destructive commands — recursive deletion, forced deletion, wildcard deletion
  • Privilege escalation — superuser usage, overly permissive chmod, chown to root
  • Credential theft — environment variable dumping, reading SSH keys, API token extraction
  • Data exfiltration — uploads to remote servers, paste services, file sharing
  • Remote code execution — piping remote content to shell interpreters, eval of URLs

Layer 2: Content Scanner

The context hook scans tool results for prompt injection attempts — embedded instructions trying to override the agent's behavior. Detected injections are stripped before reaching the LLM.

Layer 3: System Prompt Hardening

The before_agent_start hook injects security rules into every conversation, reminding the agent to never follow instructions found inside file contents or tool outputs.

Configuration

Rules are tunable via .pi/security-policy.yaml with per-pattern severity levels:

  • block — Hard stop, ask user for confirmation
  • warn — Notify user but allow execution
  • log — Silently record in audit log

View status: /security status. View audit log: /security log. Reload policy: /security reload.

Project Security Sweeps

The secure extension provides /secure to run comprehensive AI security sweeps on any project. It can also generate and install portable security protections (input sanitizers, output filters, rate limiters, CI checks).

Extending Pi

This section covers how to write your own extensions for Pi. Every extension is a TypeScript file that exports a default function receiving the ExtensionAPI.

Extension Skeleton

// ABOUTME: Second line with key details.

export default function (pi: ExtensionAPI) {
  // Register tools, commands, shortcuts, widgets, hooks

Writing Tools

Tools are LLM-callable functions. The LLM sees the tool name, description, and parameter schema, and can decide to call it.

import type { ExtensionAPI } from "@mariozechner/pi-coding-agent";
import { Type } from "@sinclair/typebox";

export default function (pi: ExtensionAPI) {
  pi.registerTool({
    name: "my_tool",
    label: "My Tool",
    description: "Does something useful.",
    parameters: Type.Object({
      input: Type.String({ description: "The input to process" }),
      verbose: Type.Optional(Type.Boolean({ description: "Show details" })),
    }),

    async execute(toolCallId, params, signal, onUpdate, ctx) {
      const { input, verbose } = params;
      const result = `Processed: ${input}`;
      return {
        content: [{ type: "text", text: result }],
      };
    },
  });
}

Key Points

  • parameters uses TypeBox for JSON Schema generation
  • execute receives: toolCallId, parsed params, AbortSignal, streaming update callback, and ExtensionContext
  • Return { content: [{ type: "text", text: "..." }] } to send results back to the LLM
  • Use signal to check for cancellation in long-running operations

Writing Commands

Slash commands are user-invoked via /name [args] in the input. They don't go through the LLM.

pi.registerCommand("greet", {
  description: "Say hello",
  handler: async (args, ctx) => {
    const name = args.trim() || "world";
    ctx.ui.notify(`Hello, ${name}!`, "info");
  },
  autocomplete: (partial) => {
    return [
      { label: "world", description: "Greet the world" },
      { label: "agent", description: "Greet the agent" },
    ].filter(item => item.label.startsWith(partial));
  },
});

Using Hooks

Hooks let you intercept and modify agent behavior at key lifecycle points.

HookWhenUse Case
tool_callBefore every tool executionBlock dangerous commands, validate inputs, add logging
contextBefore messages sent to LLMModify messages, inject context, strip content
before_agent_startBefore agent's first responseInject system prompts, configure tools
session_switchWhen switching sessionsReset state, clean up resources
session_before_compactBefore context compactionSave state, extract insights

Example: tool_call Hook

pi.registerHook("tool_call", async (event, ctx) => {
  const { name, input } = event;
  if (name === "bash" && input.command?.includes("dangerous")) {
    return { blocked: true, message: "Blocked by policy" };
  }
  return { blocked: false };
});

Example: before_agent_start Hook

pi.registerHook("before_agent_start", async (event, ctx) => {
  return { systemPrompt: "Always respond concisely." };
});

Widgets

Widgets are TUI components rendered in the terminal UI above or below the editor.

import { Text, Container } from "@mariozechner/pi-tui";

ctx.ui.setWidget("my-widget", (tui, theme) => ({
  invalidate() {},
  render() {
    return Container({
      children: [
        Text(" MY WIDGET ", { bold: true, color: theme.accent }),
        Text(" Status: running ", { color: theme.textMuted }),
      ],
    });
  },
}));
  • Widgets are identified by a string key
  • Set to undefined to remove a widget
  • Use Text, Container, Box, Spacer, Markdown from @mariozechner/pi-tui
  • Call invalidate() when widget state changes to trigger re-render

Chains & Pipelines

Creating a Chain

Add to agent/.pi/agents/agent-chain.yaml:

my-chain:
  description: "My custom workflow"
  steps:
    - agent: scout
      prompt: "Explore the codebase for: $INPUT"
    - agent: my-agent
      prompt: "Based on this analysis:\n\n$INPUT\n\nOriginal: $ORIGINAL"
    - agent: reviewer
      prompt: "Review the results:\n\n$INPUT"

Template Variables

VariableDescription
$INPUTOutput from the previous step (or user prompt for step 1)
$ORIGINALThe user's original prompt (always available)
$INPUT_NOutput from step N specifically (e.g., $INPUT_2)

Creating a Pipeline

Add to agent/.pi/agents/pipeline-team.yaml:

my-pipeline:
  description: "My custom pipeline"
  review_max_loops: 3
  phases:
    - name: understand
      description: "Clarify the task"
      mode: interactive
      agents:
        - role: scout
          task_template: "Explore and clarify: $INPUT"
    - name: build
      description: "Implement"
      mode: interactive
      agents:
        - role: builder
          task_template: "Implement: $INPUT"
    - name: review
      description: "Review"
      mode: interactive
      agents:
        - role: reviewer
          task_template: "Review: $INPUT"

Pipeline Phase Modes

  • interactive — Primary agent converses with user, may dispatch agents
  • parallel — Multiple agents dispatched concurrently
  • sequential — Single agent executes, output feeds forward
